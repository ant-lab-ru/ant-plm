## 2026-06-28 — Веха: модель данных зафиксирована (конец фазы проектирования)
**Тип:** заметка
**Контекст:** Завершили проектирование модели данных. Обе диаграммы README
(продуктовая + техническая ER) и принципы синхронны; CLAUDE.md glossary освежён.
Следующая сессия — реализация на Django (план писать по явному запросу, не сейчас).
**Итог:** Модель лот-центрична, 21 сущность, кластеризована по контурам: закупки
(`Purchase→Receipt→Lot`), производство (`Kitting`/`KittingLine`), складской ledger
(`StockMovement`), закрытие/реюз (`Writeoff`/`Inventory`/`Requisition` + внутренние
проекты). Авторство на документах (`user`/`user_id`), без ролей — осознанно.
**Открытые вопросы (не блокеры, решать при реализации):**
- **Инвентаризация и недостачи.** `Inventory` сейчас только рождает «найденные»
  партии (излишки/ре-материализация). Недостачи закрываем `Writeoff`. Если нужен
  явный пересчёт «факт vs система» с отрицательной коррекцией — додумать при дизайне
  учёта (возможно, `ADJUSTMENT`-движение).
- **Инвариант «ровно один origin»** у `Lot` (из `receipt`/`kitting`/`inventory`/
  `requisition`) — реализовать через `clean()` + DB-констрейнт.
- **Себестоимость новых лотов:** у `Requisition` `unit_cost` наследуется от исходного
  (снимок); у `Inventory`-найденных — ручная/0. Зафиксировать правило в коде.
- **Стоячие внутренние проекты** «Собственный склад» и «Свободные неучтённые» —
  как заводятся/сидируются (singletons по `Project.kind`).
- **Нумерация документов** (`number` у всех актов/УПД) — пока ручная; авто-счётчик
  при необходимости (синхронно, без Celery).
- **BOM-версии** — сознательно отложены (один BOM на изделие на старте).

## 2026-06-28 — Акты: Inventory / Writeoff / Requisition; rename Kitting
**Тип:** решение
**Контекст:** Достроили семейство складских документов и обсудили комплектование
из «Собственного склада». Два вопроса (AskUserQuestion): структура актов и как
хранить origin лота при росте с 2 до 4 типов.
**Итог:**
- **`KittingAct → Kitting`** (`KittingActLine → KittingLine`, `Lot.kitting_act_id →
  kitting_id`): ряд `Purchase/Receipt/Transfer/Kitting`, ровно одно
  самоописательное слово. README + CLAUDE.md обновлены; в журнале старое имя в
  исторических записях оставлено.
- **Структура актов — отдельные сущности** (не единый `StockAct(kind)`): добавлены
  `Inventory` (origin: «найденные» партии), `Writeoff` + `WriteoffLine` (списание с
  причиной), `Requisition` + `RequisitionLine` (требование/отпочкование). Передача
  = `Transfer` (есть). Самоописательно, печать актов прозрачна.
- **Origin лота — явные FK по типу** (не generic): добавлены `Lot.inventory_id`,
  `Lot.requisition_id` к `receipt_id`/`kitting_id`. Инвариант «ровно один origin»
  — через `clean()`/констрейнт. FK-целостность + удобные join'ы (покрытие закупки,
  генеалогия/себестоимость) важнее лаконичности на схеме такого размера.
- **Требование = отпочкование** (синк с предметником): `−qty` на исходном лоте
  (живёт дальше) + новый лот в проекте-получателе (`predecessor` = исходный). Тот
  же примитив, что закрытие, но частичный. «Постановка на баланс» = он же наоборот.
  «Вернуть остаток обратно в predecessor» сознательно не моделируем (редко).
- **Серый путь** = `Writeoff` → позже `Inventory` (с `predecessor`); **белый** =
  `Requisition` → «Собственный склад». Поддерживает спринтовое «обнуление».
**Последствия:** ER пополнен 5 сущностями (3 заголовка + 2 line), связи и принципы
README обновлены. `Inventory` снят со статуса «планируется» — он реализован в
схеме. Дальше — обновить **продуктовую** блок-схему под весь поток лота
(приход/изготовление → склад → комплектация/требование/передача → закрытие:
списание/баланс/инвентаризация). Освежить устаревший glossary в CLAUDE.md
(проект уже не nullable).

## 2026-06-28 — Проект — свойство лота; авторство на документах; закрытие через акты
**Тип:** решение (пересматривает черновик этой же сессии)
**Контекст:** Ревизия `StockMovement`. Сначала предложил `project_id` и `created_by`
**на движении**; предметник возразил, и его модель оказалась чище.
**Итог:**
- **Авторство — на документах, не на движении.** Движение не существует без
  документа (`source` обязателен), документы редки и всегда заведены сотрудником →
  автор и дата берутся из документа. `created_by` добавлен в `Purchase`/`Receipt`/
  `KittingAct`/`Transfer`, убран из `StockMovement`. Сущность `Employee → User`
  (команда, не наёмники); физически Django `auth_user`, reserved-word `USER` не
  мешает (неймспейс таблицы).
- **`project` — свойство лота (`Lot.project`, immutable), не движения.** Лот живёт
  в одном проекте от origin-документа до закрывающего; `movement.project` выводится
  из лота, межпроектного переноса одного лота **нет**. `StockMovement.project_id`
  убран. «Общее через `NULL`» заменено явными **внутренними проектами**
  (`Project.kind`): «Собственный склад» (белые/на балансе) и «Свободные неучтённые»
  (серые/списанные). `project` на документах — обязателен.
- **Закрытие проекта — закрывающими документами.** Невыбранные лоты (`qty>0`)
  сводятся в 0 актом передачи/списания/постановки на баланс. Реюз = **акт
  инвентаризации** рождает новые лоты во внутреннем проекте, `Lot.predecessor`
  хранит ссылку на закрытый лот (раньше связь была оффлайн). Команда работает
  спринтами — «обнуляется» в конце, не теряя контроль над остатками.
- Отвергнут мой «двуногий межпроектный перенос одного лота» — заменён на
  `close+recreate`, чище и точнее отражает физику «обнуления».
**Последствия:** ER: `User`, `created_by` на 4 документах, `Lot.project` +
`Lot.predecessor`, `Project.kind`; убраны `StockMovement.project_id`/`created_by`;
`project` на `Purchase`/`Receipt` стал обязательным; связи `PROJECT 1:N LOT (home)`,
`LOT 1:N LOT (predecessor)`, `USER 1:N документы`. Обновлены принципы README.
**Акт инвентаризации** (3-й origin лота + механизм реюза) теперь на критическом
пути — проектировать следующим. Продуктовую схему обновить под это (диспозиции
закрытия = передача/списание/баланс; реюз через инвентаризацию).

## 2026-06-28 — Себестоимость произведённого лота: снимок + кнопка пересчёта
**Тип:** решение
**Контекст:** Нужны были оба режима цены прибора: (а) строгая сумма стоимостей
компонентов (ради этого и затевался PLM — раньше считали руками), (б) свободная
цена поставки с добавленной оффлайн стоимостью работ. Вопрос: как ужить оба в
одном `unit_cost` без раздувания схемы.
**Итог:** Выбран **третий вариант — снимок + ручной помощник**, схему **не
трогаем**.
- `unit_cost` произведённого лота = **снимок**: префилл при создании `KittingAct`
  суммой `Σ (line.qty × component_lot.unit_cost)`, дальше статичен, правится
  руками (стоимость работ поверх материалов).
- Пересчёт — **ручная кнопка «обновить по акту изготовления»** на движке
  разузлования. Три обязательных свойства помощника (в форме, не в модели):
  **(1) один уровень** — опираться на сохранённый `unit_cost` детей, не
  перезаписывать их рекурсивно (иначе тихо сносит их ручную работу); стоимость и
  так всплывает снизу вверх. **(2)** предупреждать о компонентах без цены (иначе
  сумма молча занижена). **(3)** показывать дифф перед записью (затирает ручную
  наценку — разница и есть стоимость работ).
- Для покупных лотов (`origin=Receipt`) кнопка неактивна.
**Последствия:** Никаких новых полей. Добавлен принцип в README (семантика
`unit_cost` для произведённых лотов). Реализация: подсчёт-префилл при создании
акта + помощник пересчёта; опционально live-отчёт «себестоимость по материалам»
как проверка, не подменяя снимок.

## 2026-06-28 — Заводской номер, разузлование актов, передача по Lot
**Тип:** решение
**Контекст:** Проектирование заводского номера для производимых изделий + как
быть с многоуровневым производством (батч плат → сборка прибора) и передачей.
**Итог:**
- **`Lot.serial_number`** — ручной текст, **не ключ**, nullable, без `unique`.
  Модель «серийник = экземпляр» (`Lot` на прибор, `qty=1`), но поле текстовое →
  допускает диапазон («05–25») для батч-передач. Один `KittingAct` может породить
  как N партий-приборов (номер на каждую), так и одну партию на N — мнимый минус
  «30 приборов = 30 актов» снят (акт 1:N лотов уже в модели).
- **Область:** серийник только конечным изделиям; промежуточные партии (батч
  печатных плат) — без номера.
- **Разузлование акта изготовления = рекурсия по цепочке** `Lot →(origin)
  KittingAct →(lines) Lot → …`, тот же движок, что BOM. «Паспорт прибора» и
  объединённый «акт на прибор» — **проекция на печать, не запись в БД**. Идею
  «spawn общего акта» / self-FK `parent_act` отвергли: единственный источник
  правды — атомарные акты + цепочка лотов, дублировать нельзя (риск рассинхрона).
  Ключевой довод: промежуточную партию плат (кейс 5+25) нельзя схлопнуть — она
  реальный складской `Lot` с остатком.
- **Передача — только по `Lot`.** `Transfer` (шапка: project, date, № накладной)
  + **`TransferLine`** (`lot`, `qty>0`, `display_name?`). Убраны `Transfer.item`/
  `qty`; `item` выводится из `lot.item`. `display_name` переопределяет
  `received_name` в исходящей накладной («Прибор X. Заводские номера 05–25»).
  Передаём только готовое железо; КД и пр. — оффлайн.
- `KittingAct.qty` оставили как есть (стал denormalized, но не трогаем — без
  лишних возможностей).
**Последствия:** Обновлены обе диаграммы README (ER: `Lot.serial_number`,
`Transfer`/`TransferLine`, связи `LOT 1:N TRANSFERLINE`, `TRANSFER 1:N
TRANSFERLINE`; убрана `ITEM 1:N TRANSFER`) и принципы модели. Передача теперь
lot-центрична, как и весь склад.

## 2026-06-27 — Закупки: Purchase, покрытие через Receipt, origin партий
**Тип:** решение
**Контекст:** Финал блока закупок/партий. Три уточнения: имя `Order` конфликтует
с SQL; нужна ли двойная ссылка партии на заказ; как партия появляется не от
закупки.
**Итог:**
- Сущность заказа названа **`Purchase` / `PurchaseLine`** (вместо `Order` —
  зарезервированное слово). `Receipt.order_id → purchase_id`.
- **Убран `Lot.order_line_id`.** Партия ссылается только на `Receipt` (УПД
  синхронизирована с закупкой). Покрытие строки выводится по цепочке
  `Lot → Receipt → Purchase` + совпадению `item`; для однозначности — **уникальная
  пара `(purchase, item)` в `PurchaseLine`**. Кейс 100 = 60 + 40 работает (две
  УПД под одной закупкой → две партии). Связи стали однозначнее, потерь нет.
- **Происхождение `Lot` — всегда origin-акт, не «из воздуха»:** поставка
  (`Receipt`), изготовление (`KittingAct` = акт комплектации/изготовления, добавлен
  `Lot.kitting_act_id`), инвентаризация (акт — **планируется**). Инвариант: ровно
  один origin задан. Этим закрыт прежний открытый вопрос про самодельные партии.
**Последствия:**
- `KittingAct` теперь и потребитель (списывает компоненты), и производитель
  (создаёт партию изделия) — детали проработать при дизайне комплектации.
- Добавить сущность «Акт инвентаризации» при проработке учёта остатков.
- При реализации на Django зафиксировать имена таблиц (избежать SQL-слов);
  инвариант origin и `(purchase, item)` — через констрейнты/`clean()`.

## 2026-06-27 — Упрощение закупок: лёгкий Order вместо RFQ
**Тип:** решение
**Контекст:** RFQ/сравнение цен/счета — оффлайн, в систему не нужны. Ценность
только в формировании `order.xlsx` для поставщиков и в отслеживании «что ещё не
заказано».
**Итог:**
- Удалены `Rfq`, `RfqItem`, `RfqQuote` и тяжёлый `PurchaseOrder`/`PurchaseOrderLine`.
- Введён лёгкий **`Order`** (`project?`, `status`, `date`, `note`) + **`OrderLine`**
  (`item`, `qty`). Без поставщика и цен — они оффлайн; реальный поставщик
  появляется в `Receipt` (УПД). `order.xlsx` = экспорт заказа/дефицита.
- **Покрытие — вариант (б), точный.** `OrderLine` разрешается в один или
  несколько `Lot` через `Lot.order_line_id` (nullable). Поставка 100 = 60 + 40 =
  две партии с разными датами — штатно. Статус заказа отражает покрытие
  (draft → sent → partial → received).
- `Receipt.order_id` (бывш. `po_id`) замыкает УПД на заказ на уровне документа;
  `Lot.order_line_id` — на уровне строки. Инвариант: `lot.order_line.order ==
  lot.receipt.order`.
- **«Что ещё не заказано»** — вычисляемый отчёт, не таблица:
  `надо (BOM×ProjectDemand) − склад (Lot) − заказано (открытые OrderLine)`.
**Последствия:** Минус 5 сущностей. Обновлены обе диаграммы (продуктовая:
`дефицит → Order → УПД`; техническая: Order/OrderLine, связь OrderLine 1:N Lot) и
принципы модели в README.

## 2026-06-27 — Lot-центричный разворот склада
**Тип:** решение
**Контекст:** Разбирали разрыв «абстрактный Item (PartNumber из Altium/BOM) ↔
реальная позиция закупки с названием из УПД». Рассматривали ввод `SupplierPart`
(Vendor Part), но отказались: заказы у поставщиков уникальны (каждый раз новый
поиск/поставка), повторного заказа одной позиции нет — значит каждый приход даёт
новый уникальный `Lot`. Тащить заводскую «классику» (каталоги поставщиков, история
моделей) не нужно — цель лёгкий кастомный PLM.
**Итог:** `Lot` сделан главной учётной единицей:
- `Lot` хранит `unit_cost` (цена закупки) и `received_name` (название из УПД).
- Поставщик и дата НЕ дублируются в `Lot` — берутся через `Receipt` (нормализация).
- `ReceiptLine` **удалён**: связь приход↔партия 1:1, заменена на `Lot.receipt_id`
  (`Receipt` 1:* `Lot`). `receipt_id` **nullable** — задел под самодельные партии
  (собранный узел как Lot без УПД); это решение по умолчанию, можно ужесточить.
- `StockMovement` двигается **только по `Lot`**: поле `item_id` убрано (выводится
  из `lot.item_id`). Добавлен тип `RETURN` (возврат с пайки).
- `lot_number` (заводской batch-код) удалён как избыточный для НИР/мелкосерии;
  идентичность партии несёт `received_name`.
- Начальное количество партии = первое движение `RECEIPT` (+qty), отдельного поля
  qty в `Lot` нет.
**Последствия:** Сущность `SupplierPart` не вводим. Печать акта комплектации
тянет реальное название через `KittingActLine.lot → received_name`, а связь с BOM
— через `component`. Обновлены обе диаграммы и принципы модели в README.
**Открытый вопрос:** становится ли собранный образец/узел складским `Lot`
(влияет на обязательность `receipt_id`) — пока nullable.

## 2026-06-27 — Уточнение модели: единый Item, передача, остатки
**Тип:** решение
**Контекст:** Ответы предметника по развилкам Item/передача/балансы + запрос на
визуализацию модели.
**Итог:**
- **`Item` единый** для изделий и компонентов (изделие может состоять из изделий).
- **Передача — только заказчикам**, без отдельной сущности получателя: `Transfer`
  в рамках проекта по накладной; внешний документооборот вне PLM.
- **Остатки в разрезе проекта** (`project` nullable в ledger; NULL = общее
  пользование). При закрытии проекта судьба остатков — списать / в общее / отдать
  заказчику / на баланс под будущую комплектацию — это новые движения ledger.
- Главные вопросы пользователя к системе: сколько конкретного компонента всего и
  по проектам; хватает ли всех Item на сборку изделия; что делать с остатком в

## 2026-06-27 — Базовые решения доменной модели
**Тип:** решение
**Контекст:** Старт проектирования модели данных. Уточнения от предметника по
трём развилкам, влияющим на схему.
**Итог:**
- **Изделие = один BOM** (без явных версий на старте). Строки состава держим
  отдельной таблицей `BomLine`, чтобы версии добавились позже без ломки.
- **Склад = ledger + партии + места.** Неизменяемые движения `StockMovement`;
  остаток = SUM(qty) в разрезе item / партия (`Lot`) / место (`Location`).
- **Связь проект↔склад — гибрид:** общий склад + проектная аллокация
  одновременно (у движения склада есть nullable FK на проект).
**Последствия:** Сверку балансов строим как отчёт поверх ledger в разрезе
проекта (закуплено/скомплектовано/передано/остаток), а не как отдельный мутабельный
счётчик. Многоуровневый BOM = рекурсивный обход `BomLine`.

## 2026-06-27 — Фреймворк бэкенда и БД: Django + DRF на MySQL
**Тип:** решение
**Контекст:** Стек — Python; домен насыщен связанными сущностями, ролями,

  BOM, RFQ, пересчёт балансов) синхронно; периодика — через cron. Без realtime.
